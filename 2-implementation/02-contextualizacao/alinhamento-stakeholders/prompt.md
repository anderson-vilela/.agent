# Prompts para Alinhamento com Stakeholders

## Mapeamento de Stakeholders

```
Como Gerente de Projeto experiente em engajamento de stakeholders, preciso da sua ajuda para mapear todos os stakeholders relevantes para a seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Por favor, forneça:

1. Uma lista completa de todos os stakeholders que deveriam ser envolvidos, agrupados por categorias (ex: negócio, técnico, operações, usuários, etc.)
2. Para cada stakeholder identificado:
   - Seu papel e interesse específico nesta implementação
   - Nível de influência e impacto (alto/médio/baixo)
   - Tipo de engajamento necessário (informar, consultar, obter aprovação, etc.)
   - Frequência ideal de comunicação
3. Uma matriz de poder/interesse que classifique visualmente os stakeholders
4. Recomendações sobre como priorizar o engajamento com diferentes stakeholders

Estruture sua resposta de forma que possa ser usada como base para um plano de comunicação e engajamento durante todo o ciclo de vida da implementação.
```

## Validação com Product Owner

```
Como Tech Lead preparando-se para uma reunião crucial com o Product Owner, preciso da sua ajuda para estruturar a validação dos requisitos da seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Por favor, elabore:

1. Uma agenda estruturada para a reunião de validação, incluindo:
   - Pontos-chave que precisam ser confirmados
   - Perguntas específicas para esclarecer áreas de ambiguidade
   - Demonstração do seu entendimento atual dos requisitos
2. Uma lista de tópicos específicos para validar:
   - Prioridades entre diferentes aspectos da implementação
   - Critérios de aceitação detalhados
   - Trade-offs técnicos que precisam ser discutidos
3. Um formato para documentar decisões e esclarecimentos
4. Sugestões para lidar com potenciais cenários desafiadores (ex: requisitos conflitantes, escopo inflacionado)

O objetivo é garantir um entendimento completo e alinhado com o Product Owner antes de avançar com detalhes técnicos da implementação.
```

## Alinhamento com Equipe de QA

```
Como Desenvolvedor Líder preparando-se para alinhar expectativas com a equipe de QA, preciso da sua ajuda para estruturar uma discussão produtiva sobre a seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Por favor, ajude-me a preparar:

1. Uma apresentação concisa da implementação do ponto de vista de testabilidade, incluindo:
   - Funcionalidades-chave e como elas deveriam ser validadas
   - Interfaces e integrações que precisarão de teste especial
   - Comportamentos esperados em diferentes cenários
2. Uma lista de perguntas para discutir com a equipe de QA:
   - Abordagens de teste recomendadas para esta implementação
   - Dados e ambientes de teste necessários
   - Automação de teste possível e cobertura esperada
3. Um framework para desenvolvimento colaborativo de casos de teste, incluindo:
   - Categorias de testes (funcionais, não-funcionais, edge cases, etc.)
   - Priorização baseada em risco
   - Critérios de aceitação detalhados para cada aspecto crítico

O objetivo é estabelecer uma parceria efetiva com QA desde o início da implementação, garantindo alta qualidade e prevenção de problemas.
```

## Alinhamento Técnico com Arquitetos

```
Como Desenvolvedor preparando-se para um alinhamento técnico com Arquitetos e Tech Leads, preciso da sua ajuda para estruturar uma discussão eficaz sobre a seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Por favor, ajude-me a preparar:

1. Uma apresentação técnica concisa da abordagem proposta, incluindo:
   - Arquitetura de alto nível e padrões de design considerados
   - Justificativa para decisões técnicas significativas
   - Alternativas consideradas e razões para as escolhas feitas
2. Tópicos específicos para discussão e validação:
   - Aderência aos padrões arquiteturais do sistema
   - Performance, escalabilidade e considerações de segurança
   - Integração com outros componentes e serviços
3. Um formato para documentar as decisões de arquitetura (ADR), incluindo:
   - Contexto e declaração do problema
   - Opções consideradas
   - Decisão e justificativa
   - Consequências e trade-offs

O objetivo é obter alinhamento técnico e aprovação da abordagem proposta, além de capturar feedback valioso dos arquitetos e líderes técnicos antes de prosseguir com a implementação detalhada.
```

## Plano de Comunicação Contínua

```
Como Project Manager com forte experiência em comunicação efetiva, preciso da sua ajuda para criar um plano de comunicação contínua para a seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Considerando os diversos stakeholders envolvidos:

[LISTE OS PRINCIPAIS GRUPOS DE STAKEHOLDERS]

Por favor, desenvolva um plano de comunicação que inclua:

1. Estratégia de comunicação, definindo:
   - Canais de comunicação preferidos para diferentes tipos de atualizações
   - Frequência e formato de comunicações regulares
   - Gatilhos para comunicações especiais ou extraordinárias
2. Matriz de comunicação detalhando:
   - Que informação será compartilhada com quais stakeholders
   - Quem é responsável por cada tipo de comunicação
   - Como feedback será coletado e incorporado
3. Templates para diferentes tipos de comunicação:
   - Atualizações de status regulares
   - Comunicação de problemas ou bloqueios
   - Solicitações de decisão ou input
4. Plano de escalação para problemas ou decisões críticas

O plano deve ser prático, eficiente e focado em manter todos os stakeholders apropriadamente informados e engajados durante toda a implementação.
```

## Documentação de Alinhamento e Decisões

```
Como Analista de Negócios especializado em documentação técnica, preciso da sua ajuda para criar um template para documentar alinhamentos e decisões para a seguinte implementação:

[DESCREVA BREVEMENTE A FUNCIONALIDADE/COMPONENTE A SER IMPLEMENTADO]

Por favor, desenvolva:

1. Um template de documento de alinhamento que capture:
   - Sumário executivo das decisões-chave e alinhamentos
   - Stakeholders envolvidos e seus papéis
   - Questões discutidas e resolvidas
   - Decisões tomadas, com justificativas
   - Próximos passos e responsabilidades
2. Um formato leve de ADR (Architecture Decision Record) para decisões técnicas, incluindo:
   - Título e ID da decisão
   - Contexto e problema sendo resolvido
   - Opções consideradas
   - Decisão tomada e justificativa
   - Consequências (positivas e negativas)
   - Status e data
3. Uma estrutura para registrar questões em aberto que precisam de acompanhamento, incluindo:
   - Descrição da questão
   - Impacto se não for resolvida
   - Responsável pela resolução
   - Data limite
   - Status atual

Este template deve ser leve o suficiente para uso prático, mas abrangente o suficiente para servir como registro formal de alinhamentos e decisões críticas.
```
